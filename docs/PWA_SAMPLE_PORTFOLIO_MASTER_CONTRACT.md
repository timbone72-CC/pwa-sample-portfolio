# PWA Sample Portfolio Master Contract

## 1. Purpose

This project exists to turn proven FieldLedger-style work into a focused portfolio of lightweight small-business PWA samples.

The goal is to show working, practical apps that can be built quickly for small businesses, 1099 workers, field crews, office teams, and owner-operators.

FieldLedger remains the flagship proof-of-work and the closest example of a real completed product.

## 2. Relationship to FieldLedger

FieldLedger is not abandoned or hidden from this project.

FieldLedger may be used as:

- a flagship case study
- design inspiration
- proof of offline-first PWA capability
- proof of Google Sheets workflow capability
- proof of printable reports, CSV export, JSON backup, and mobile-first design

FieldLedger must not become the dumping ground for unfinished sample-app experiments.

Reusable ideas may be extracted from FieldLedger, but sample apps should live in their own project lane.

## 3. Portfolio Project Goal

Build a set of polished sample PWAs that demonstrate reusable small-business workflows:

- quick job trackers
- timesheets
- invoice helpers
- field-work logs
- Google Sheets workflows
- printable reports
- CSV exports
- JSON backups
- lightweight offline-first operation

Each sample should be simple enough to understand quickly and polished enough to show publicly.

## 4. Public and Private Repo Rules

Private repos are for:

- unfinished work
- reusable templates
- contracts
- experiments
- client-style planning
- rough ideas
- prompts
- internal business strategy

Public repos are for:

- polished sample apps
- live demos
- clean README files
- screenshots
- case studies
- finished portfolio material

No unfinished sample app should be made public until it is useful, understandable, and honest about what works.

## 5. First Sample App

The first sample app will be:

**Job Tracker PWA**

Reason:

- closest to FieldLedger
- easiest to explain to small businesses
- useful across many industries
- strong fit for mobile workers
- good base for reusable record/report/export patterns

## 6. Shared Template Strategy

The sample apps should be designed using reusable layers.

### 6.1 Core PWA Shell

Shared across all apps:

- mobile-friendly layout
- dashboard
- navigation
- settings
- help page
- local storage
- offline-first behavior
- JSON backup/restore
- print styling
- future add-on preview cards

### 6.2 Shared Feature Modules

Reusable where appropriate:

- record forms
- saved record lists
- status badges
- printable reports
- CSV export
- JSON backup
- photo attachment
- notes sections
- totals summaries
- help/instruction panels

### 6.3 App-Specific Workflow

Changed per app:

- field names
- business rules
- report wording
- export columns
- calculations
- labels
- setup instructions

## 7. Google Sheets Rule

Google Sheets integration is optional.

If a sample app clearly needs Google Sheets, the Sheets workflow may be built as part of that sample.

If the app does not clearly need Sheets, Sheets remains a visible optional add-on or later phase.

Sheets must not be forced into every sample app.

## 8. Future Feature Preview Rule

Sample PWAs may show future-feature previews when useful.

Future features must be clearly labeled as:

- Future Add-On
- Planned
- Optional Module
- Not included in this starter version

No sample app should pretend that inactive features are working.

## 9. Build Order

### 9.1 Phase 0 — Planning Contracts

Create the planning contract set before building new sample apps.

### 9.2 Phase 1 — FieldLedger Case Study

Prepare FieldLedger as the flagship proof-of-work.

### 9.3 Phase 2 — Job Tracker PWA Sample

Build the first public-ready sample app.

### 9.4 Phase 3 — Shared Template Extraction

Identify which Job Tracker sections should become reusable templates.

### 9.5 Phase 4 — Timesheet PWA Sample

Reuse the shared shell and record/report/export modules.

### 9.6 Phase 5 — Invoice Helper PWA Sample

Reuse the shared shell and report/print/export modules.

### 9.7 Phase 6 — Field Work Log PWA Sample

Reuse the shared shell, records, notes, photos, and reports.

### 9.8 Phase 7 — Google Sheets Workflow Sample

Build or extract a reusable Sheets workflow only after the simpler samples are stable.

## 10. Stop Conditions

Do not build new apps if:

- FieldLedger case study is unclear
- the first sample is unfinished
- reusable parts are not documented
- public/private repo rules are unclear
- the next app does not reuse anything from the last app

## 11. Current Decision

The current next step is to create this master contract first.

No app code should be written until this contract is accepted.
